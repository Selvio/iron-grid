# Server database layer

Server-only PostgreSQL 17 + Drizzle ORM persistence (`docs/03-architecture/database.md`).
Only the backend imports this module; `game-engine` and `game-data` never do
(`architecture.md` §4, enforced by `__tests__/forbidden-import.test.ts`).

## Layout

- `schema/` — one Drizzle table slice per file, re-exported from `schema/index.ts`.
- `queries/` — the persistence primitives (snapshot write, event append,
  idempotency, concurrency, version pinning). No HTTP/pipeline logic — that is M7.
- `client.ts` — the pooled Neon client; `env.ts` — `DATABASE_URL` access.
- `__tests__/` — PGlite-backed integration tests and the layer guards.

## Migrations

Generated by Drizzle Kit into `drizzle/` and checked in. Regenerate after any
schema change:

```
pnpm db:generate   # emit SQL from the schema diff
pnpm db:migrate    # apply forward-only (needs DATABASE_URL)
```

**Integrity rule** (`database.md` §9, `data_versioning.migration_of_active_match`):
migrations are **forward-only** and **additive**. A migration may add columns or
tables, but must **not** rewrite the semantics of an active match's `state`
snapshot or its pinned `game_data_version`. A structural change that would alter
how an in-flight match is interpreted requires an explicit administrative
migration, never an in-place edit.

## Test harness boundary

Schema shape, constraints, migration apply and version logic are tested against
in-process **PGlite** (`__tests__/harness.ts`). PGlite is single-connection, so
it cannot prove true `FOR UPDATE` contention; the one two-connection row-lock
serialization test uses a real multi-connection Postgres
(`m4-persistence.md` §3).
